Introduction

The main goal of this guide is to help developers on common simple and complex use cases developing an application with Angular.

Is a very opinionated guide and the aim is at high level. This isn't a from zero guide nor an API Reference. Official Angular documentation will be referenced in such cases when some basic knowledge is needed.

About Angular

Angular is a web application develoment framework to be used for the Front-End. It was developed to run on Javascript. However it's desgined to be used in Typescript, and transpiled to Javascript or other targets.

I say that it is a Web Application Framework for a reason. If you want to create a mockup, a static website or a small app (small enough to be only maintained by you) Angular will be an overkill and there are other frameworks out there that may fit your needs more accurately (like React for example).

Web Components vs Web Applications

Other frameworks like React are Web Component Frameworks, but Angular is a Web Application framework because it comes with a sort of other tools (and opinions) that go far beyond Web Components.

Ok, Angular helps creating components, but:

  • You can't render a component outside Angular (before Ivy). There is Angular Elements, but it still works inside Angular
  • It does a LOT of more things for you: building proccess handling, bundling, routing, lazy loading, platform agnostic rendering, tooling for state management, i18n handling, component prototyping

So, we can say that Angular is a Web Component framework, but it also adds a lot of functionality that you are going to need for developing a full Web Application.

In the end, if your application is big and serious enough, you are going to need this kind of tools. The difference is Angular has made the decision for you already.

React for example started as a simple library for Front-End components, but now you can find a lot of tools in the React environment that suit the roles of the rest of parts of Angular (because in the end, you are going to need them).

That is why I say that Angular (as this guide) is very opinionated. You can review and decide what tool or strategy use for each situation and edge case, or go ahead with the solution more experienced developers proposed

This, however, isn't always a pro. Sometimes we need something more specific or handmade tailored.

Anyway, everything can be changed. The Angular team deeply though about it and they improve the framework with each release, so everything is abstracted enough to be replaced with custom parts when needed.

What's inside Angular?

So, let's take a quick look to what comes with Angular:

  • Angular CLI
  • ngc - Ng Compiler (a wrapper around Typescript Compiler)
  • Webpack - Bundler
  • Zone.js - Asynchronus abstraction layer (asynchronus context provider, change detection, error handling)
  • RxJS - Observer pattern library/framework
  • Karma - Test runner
  • The Angular Core - Renderer2 and rest of abstractions
  • WIP?

The Angular Core it's Angular itself, and it relays on RxJS for every reactive interaction, but besides that, any other part of the package could be changed to another tool.

Angular CLI

Ok, so we have a lot of different packages, libraries and tools being use there in Angular. How does it orchestrates them? Using the CLI.

ng (Angular CLI) is the Angular's swiss army knife.

Angular CLI anatomy

Well, let's start describing it and its usage.

First of all, the Angular CLI is no more than the glue for other tools outside the Angular CLI and the dependencies of itself.

These dependencies are the real tools doing the job on the background. So we have something like this:

  • @angular/cli - Main package
  • @schematics/angular
  • @schematics/update
  • @angular-devkit/core
  • @angular-devkit/architect
  • @angular-devkit/schematics

Let's start with the glue itself.

@angular/cli handles the comands, default behaviors and the external tools used in the build/deploy process (Webpack, Karma, TsLint...). In summary, it handles our Angular Workspace and the automatic tasks run inside it.

Ok, quite straight forward isn't it?

Then we have the packages that are part of the @schematics/* namespace. These contain the default schematics provided by Angular Team, and used through the ng generate command.

And the last part: @angular-devkit/*

These packages are the engine powering Angular CLI on the background.

@angular-devkit/core: Utility functions used by the rest of packages of @angular-devkit

@angular-devkit/schematics: The engine for creating code from schematics.

@angular-devkit/architect: Engine for complex build tools (we'll see it more in-depth later).

Angular CLI Key Concepts

Angular Workspace

So, I've said that Angular CLI is the handler of your Angular Workspace, but what is exactly the worksapce and what does this mean?

Simplifying A LOT, the workspace is the root folder of your Angular project. Well, at least if you create it via ng new (using the Angular CLI).

This folder contains every file needed by the Angular CLI to run any operation over the project, from creating code to deploying the application.

You can find the official definition of what is an Angular Workspace on the Angular documentation glossary.

Schematics

Schematics are a series of scripts (usually contained in a library) that can be run to automate specific actions over the filesystem.

Their role is usually to generate code.

Angular CLI contains a set of default schematics used by its command ng generate

ng generate component shared/components/my-shared-component

Angular CLI comes with a variety of schematics to generate different kind of code: applications, components, services...

As we'll see later, you can also create your own schematics.

Architects & Builders

Architects & Builders are two concepts that work together for manage our code from Angular CLI

The builders do operations over our code (build the project, serve it, deploy it, run tests...). Each builder is normally related to a dedicated npm package performing a specific task.

The architects map between the CLI, the targets & the builders, and also allow to configure the builder for the specific target.

So, more or less, architects parse commands to determine the builder, the target and the additional configuration to be used, and then the builders do the real job.

Angular.json

The angular.json file is the file previously known as angular-cli.json

Here is where we link all the pieces of the workspace together so they can work.

All the configuration for the workspace, its projects, its architects and builders is defined in this file.
